This diagram shows one possible schema for storing OpenAlex data in a [relational database](/download/relational-database). It's the one used in our PostgreSQL examples, but may not be the best layout for your specific queries.

The schema flattens nested JSON objects into separate tables with foreign key relationships. For example, a work's authorships become rows in a `works_authorships` table that references both `works` and `authors`.

## Key design decisions

- **One main table per entity type** (`works`, `authors`, `sources`, `institutions`, `topics`, `publishers`, `funders`)
- **Separate tables for nested arrays** (e.g., `works_authorships`, `works_open_access`, `concepts_related_concepts`)

- **Indexes on foreign keys** for efficient joins
